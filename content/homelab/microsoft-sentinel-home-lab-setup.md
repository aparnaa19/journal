<p><strong>Home Lab Setup: Microsoft Sentinel</strong></p><p><strong>Overview</strong></p><p>In this lab, I set up Microsoft Sentinel as my SIEM environment on Azure. This is the home lab foundation for my Impossible Travel / Account Compromise investigation. The goal of this setup phase was to get Sentinel running, ingest custom login logs, and verify the data is queryable before moving into detection.</p><p><strong>Why Microsoft Sentinel?</strong></p><p>In this lab, I chose Microsoft Sentinel because impossible travel detection is fundamentally an identity and authentication log problem — not an endpoint problem. Sentinel is cloud-native, built around log ingestion and KQL-based detection, and has industry recognition among SOC teams. It also offered a free 31-day trial which made it accessible for a home lab setup.</p><p><strong>What I Set Up</strong></p><p>In this lab, the environment has three layers:</p><ul><li><strong>Log Analytics Workspace</strong> — this is where logs are stored and queried</li><li><strong>Microsoft Sentinel</strong> — the SIEM that sits on top and handles detection and incidents</li><li><strong>Custom Log Table (ImpossibleTravel_CL)</strong> — where my synthetic dataset lives</li></ul><p><strong>Step 1 — Creating the Azure Account</strong></p><p>In this step, I created a free Azure account at azure.microsoft.com/free using my personal Gmail. I chose personal email over my college email because college accounts lose access after graduation.</p><p>Azure gives $200 in free credits for 30 days. A credit card is required for identity verification but is not charged unless you manually upgrade.</p><p><strong><img src="media/microsoft-sentinel-home-lab-setup-1.webp" /></strong></p><p><strong>Step 2 — Creating the Log Analytics Workspace</strong></p><p>In this step, I created the Log Analytics Workspace which acts as the database for all logs that Sentinel will analyze. Without this, Sentinel has nowhere to store or query data.</p><p>I navigated to Log Analytics Workspaces → Create and used these settings:</p><ul><li><strong>Resource Group:</strong> SOC-Lab</li><li><strong>Workspace Name:</strong> SOC-Lab</li><li><strong>Region:</strong> East US</li></ul><p><img src="media/microsoft-sentinel-home-lab-setup-2.webp" /></p><p><strong>Step 3 — Setting Up a Cost Alert</strong></p><p>In this step, I set up a $1 budget alert so I get an email if anything unexpected gets charged. This is good practice before touching any cloud billing settings. It takes 2 minutes and gives complete peace of mind.</p><p><img src="media/microsoft-sentinel-home-lab-setup-3.webp" /></p><p><strong>Step 4 — Enabling Microsoft Sentinel</strong></p><p>In this step, I enabled Sentinel on top of the SOC-Lab workspace. Sentinel adds the SIEM layer — analytics rules, incident management, and threat hunting on top of the raw logs.</p><p>I navigated to Microsoft Sentinel → Create → selected SOC-Lab → clicked Add.</p><p>The free trial kicked in automatically:</p><ul><li><strong>Free period:</strong> 6/11/2026 → 7/12/2026</li><li><strong>Free ingestion:</strong> up to 10 GB/day</li><li><strong>Cost for our lab:</strong> effectively $0 (our dataset is 43 rows / ~15KB)</li></ul><p><img src="media/microsoft-sentinel-home-lab-setup-4.webp" /></p><p><strong>Step 5 — Getting Workspace Credentials</strong></p><p>In this step, I retrieved the Workspace ID and Primary Key needed to ingest logs programmatically via the HTTP Data Collector API.</p><p>The Workspace ID was available in Settings → Properties. The Primary Key however was not visible in the Azure portal UI — Microsoft removed it in the newer portal version. I retrieved it using Azure Cloud Shell instead by running:</p><p>az monitor log-analytics workspace get-shared-keys \</p><p>  --resource-group SOC-Lab \</p><p>  --workspace-name SOC-Lab \</p><p>  --query primarySharedKey -o tsv</p><p><img src="media/microsoft-sentinel-home-lab-setup-5.jpeg" /></p><p><strong>What I Learned</strong></p><p>In this setup phase, I learned that Microsoft Sentinel is not a standalone tool — it's a layer that sits on top of Log Analytics. The workspace is the real engine; Sentinel is the interface. I also learned that Azure's newer portal removed some legacy features like the visible Primary Key, which meant I had to use CLI to retrieve it — something I wouldn't have encountered just reading documentation.</p><p><img src="media/microsoft-sentinel-home-lab-setup-6.webp" /></p><p><strong>Environment at a Glance</strong></p><table><thead><tr><th><p><strong>Component</strong></p></th><th><p><strong>Value</strong></p></th></tr></thead><tbody><tr><td><p>Azure Account</p></td><td><p>aparnaa19.08@gmail.com</p></td></tr><tr><td><p>Resource Group</p></td><td><p>SOC-Lab — East US</p></td></tr><tr><td><p>Log Analytics Workspace</p></td><td><p>SOC-Lab</p></td></tr><tr><td><p>Sentinel Trial</p></td><td><p>Active until 7/12/2026</p></td></tr><tr><td><p>Budget Alert</p></td><td><p>$1 threshold</p></td></tr></tbody></table>
