<p><strong>Wazuh SIEM Home Lab  </strong>[Wazuh XDR + Windows Agent Setup]</p><p><strong>1. Lab Overview</strong></p><p>This home lab sets up Wazuh as a SIEM and XDR platform for investigating a Windows brute force attack. The environment runs entirely on a local Windows 11 machine using Docker. A Wazuh agent is installed on the same machine, configured to forward logs to the Wazuh Manager running in Docker containers, and a pre-recorded EVTX brute force dataset is ingested for investigation.</p><p>Wazuh is an agent-based platform - every monitored endpoint runs a lightweight agent that collects logs, performs local integrity checks, and ships telemetry to a central Manager. This is the same model used by commercial EDR/SIEM products like CrowdStrike and Splunk. Setting it up from scratch, including troubleshooting real configuration errors, is the kind of experience that a tutorial alone cannot replicate.</p><p><strong>2. Setup Steps</strong></p><p><strong>Step 1 - Clone the Wazuh Docker Repository</strong></p><p>Wazuh provides an official Docker Compose configuration. The repository is cloned at the v4.12.0 tag to ensure all three components are version-consistent. The single-node folder is used because all components run on one machine.</p><pre data-lang="bash"><code>git clone https://github.com/wazuh/wazuh-docker.git -b v4.12.0
cd wazuh-docker/single-node
ls</code></pre><p>The single-node folder contains docker-compose.yml (defines all three containers), generate-indexer-certs.yml (certificate generation), and a config directory with component configuration files.</p><p><em><img src="media/wazuh-siem-home-lab-2.png" /></em></p><p><strong>Step 2 - Resolve Port Conflicts</strong></p><p>If Wazuh Indexer runs on port 9200 by default, which contains some conflicts – for example with Elasticsearch if both are running on the same machine. The Wazuh Indexer external port is remapped in docker-compose.yml.</p><p>Find the indexer ports section and change “9200:9200” to “9201:9200’</p><p>This maps port 9201 on the host to port 9200 inside the container. Wazuh components still communicate internally on 9200 - only the external port changes. Elasticsearch continues to use 9200 without conflict.</p><p>Port mapping format is <em>host_port:container_port</em>. Changing the left side only affects what port is exposed on the Windows machine - everything inside Docker stays the same.</p><p><strong>Step 3 - Generate SSL Certificates</strong></p><p>Wazuh components communicate over TLS, so certificates must be generated before containers start. This runs once and exits automatically.</p><p>docker compose -f generate-indexer-certs.yml run --rm generator</p><p>The output confirms certificates are created for the root CA, admin, indexer, filebeat, and dashboard. The container exits on its own when done.</p><p><strong>Step 4 - Start Wazuh</strong></p><pre><code>docker compose up -d</code></pre><p># Verify all three containers are running:</p><pre><code>docker ps</code></pre><p>Expected outputs is: </p><pre><code># single-node-wazuh.manager-1    Up
# single-node-wazuh.indexer-1    Up
# single-node-wazuh.dashboard-1  Up</code></pre><p><strong>Step 5 - Access the Wazuh Dashboard</strong></p><p>Open a browser and navigate to https://localhost:443. A security warning appears because the certificate is self-signed - click Advanced and proceed.</p><p>Credentials are stored in docker-compose.yml. Find them by running:</p><pre><code>Select-String -Path ".\docker-compose.yml" -Pattern "USERNAME|PASSWORD"</code></pre><p><em><img src="media/wazuh-siem-home-lab-3.png" /></em></p><p><strong>Step 6 - Install the Wazuh Agent</strong></p><p>The agent version must match the Manager version exactly. A newer agent will be rejected at registration. The 4.12.0 MSI is installed silently via PowerShell as Administrator:</p><pre><code>msiexec /i "E:\path\to\wazuh-agent-4.12.0-1.msi" WAZUH_MANAGER="127.0.0.1" /qn</code></pre><p># Verify executables are present:</p><pre><code>ls "C:\Program Files (x86)\ossec-agent\*.exe"</code></pre><p>Expected files:</p><pre><code>agent-auth.exe    #handles registration with the Manager
wazuh-agent.exe   #the main agent service
manage_agents.exe #agent management utility</code></pre><p><img src="media/wazuh-siem-home-lab-4.png" /></p><p><strong>Step 7 - Register the Agent</strong></p><pre><code>cd "C:\Program Files (x86)\ossec-agent"
.\agent-auth.exe -m 127.0.0.1</code></pre><p>Expected output:</p><p># INFO: Requesting a key from server: 127.0.0.1</p><p># INFO: Valid key received</p><p>If registration fails with "Duplicate agent name", the old agent entry must be removed from the Manager first:</p><pre><code>docker exec -it single-node-wazuh.manager-1 /var/ossec/bin/manage_agents -r 001</code></pre><p>Enter R, provide the agent ID, confirm with y</p><p>Then re-run agent-auth.exe.</p><p><em><img src="media/wazuh-siem-home-lab-5.png" /></em></p><p><strong>Step 8 - Configure ossec.conf</strong></p><p>The ossec.conf file is at C:\Program Files (x86)\ossec-agent\ossec.conf. Open it in Notepad as Administrator. Three changes are needed:</p><p><strong>8a. Fix the Manager address:</strong></p><pre><code>&lt;client&gt;
  &lt;server&gt;
    &lt;address&gt;127.0.0.1&lt;/address&gt;
    &lt;port&gt;1514&lt;/port&gt;
    &lt;protocol&gt;tcp&lt;/protocol&gt;
  &lt;/server&gt;
&lt;/client&gt;</code></pre><p><strong>8b. Remove unsupported syscollector tags (not supported in 4.12.0):</strong></p><pre><code>&lt;!-- Remove these from &lt;wodle name='syscollector'&gt; --&gt;
&lt;users&gt;yes&lt;/users&gt;
&lt;groups&gt;yes&lt;/groups&gt;
&lt;services&gt;yes&lt;/services&gt;
&lt;browser_extensions&gt;yes&lt;/browser_extensions&gt;</code></pre><p><strong>8c. Add the brute force log file as a monitored source:</strong></p><pre><code>&lt;!-- Add after the last &lt;/localfile&gt; block --&gt;
&lt;localfile&gt;
  &lt;location&gt;C:\bruteforce-events.log&lt;/location&gt;
  &lt;log_format&gt;syslog&lt;/log_format&gt;
&lt;/localfile&gt;</code></pre><p><strong>Step 9 - Ingest the EVTX Dataset</strong></p><p>Wazuh cannot subscribe to static EVTX files directly. Events are extracted using PowerShell and written to the plain log file the agent monitors:</p><pre><code>$evtxPath = "E:\appu project\identity and access\failed login investiigation\EVTX-to-MITRE-Attack-master\TA0006-Credential Access\T1110.xxx-Brut force\ID 4776,4625-AccountRestore local bruteforce.evtx"
$events = Get-WinEvent -Path $evtxPath -ErrorAction SilentlyContinue
foreach ($event in $events) {
    $log = @{
        TimeCreated = $event.TimeCreated.ToString("o")
        EventID     = $event.Id
        Message     = $event.Message
        Computer    = $event.MachineName
    } | ConvertTo-Json -Compress
    Add-Content -Path "C:\bruteforce-events.log" -Value $log
}</code></pre><p><strong>Step 10 - Start the Agent and Verify</strong></p><pre><code>NET START WazuhSvc</code></pre><p># Check agent log for connection confirmation:</p><pre><code>Select-String -Path "C:\Program Files (x86)\ossec-agent\ossec.log" -Pattern "Connected|Analyzing" | Select-Object -Last 10</code></pre><p><em><img src="media/wazuh-siem-home-lab-6.png" /></em></p><p><strong>6. Restarting the Lab</strong></p><p>Step 1: Open Docker Desktop and wait for Running status</p><p>Step 2: Start Wazuh containers</p><pre><code>cd "E:\softwares and tools\wazuh\wazuh-docker\single-node"
docker compose up -d</code></pre><p>Step 3: Wait ~60 seconds for full initialisation</p><p>Step 4: Start the agent (PowerShell as Administrator)</p><pre><code>NET START WazuhSvc</code></pre><p>Step 5: Open https://localhost:443 in browser</p><p>Previously ingested events persist in the Wazuh Indexer across restarts. Re-ingestion is only needed if the indexer volume was deleted.</p><p><strong>7.
