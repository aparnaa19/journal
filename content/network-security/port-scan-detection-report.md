# Port Scan Detection Report
display_title: Port Scan Detection Report
skills: Port Scan Detection, Network Flow Analysis, KQL Query Writing, Log Ingestion via API, T1046 Investigation
tools: Microsoft Sentinel, Azure Log Analytics, Azure Cloud Shell, KQL, Log Analytics HTTP Data Collector API

<p>Severity: medium</p><p>MITRE ATT&amp;CK T1046 - Network Service Discovery</p><p><strong>1. Executive Summary</strong></p><p>A port scan attack was detected and confirmed through analysis of 5,000 network flow records in Microsoft Sentinel. The attacker probed 998 unique destination ports with an average connection duration of 28 milliseconds per probe - consistent with automated scanning tool behavior such as Nmap or Masscan. This maps to MITRE ATT&amp;CK T1046 (Network Service Discovery), a reconnaissance technique used by attackers to identify open services before launching an exploitation attempt.</p><p><strong>2. Detection</strong></p><pre><code>PortScanCICIDS_CL
| summarize PortsScanned = dcount(Destination_Port_d),
            AvgDuration = avg(Flow_Duration_d),
            TotalFlows = count()
| extend Verdict = iif(PortsScanned &gt; 100, "CONFIRMED PORT SCAN", "Normal")
| project PortsScanned, TotalFlows, AvgDuration, Verdict</code></pre><p>This query pulls all records from the ingested dataset and counts how many distinct destination ports were hit, the average duration of each connection, and the total number of flows. It then applies a rule - if more than 100 unique ports were scanned, it labels the activity as a confirmed port scan. A legitimate application would never need to contact hundreds of different ports; only a scanning tool would.</p><p><img src="media/port-scan-detection-report-1.webp" /></p><p><strong>3. Investigation Findings</strong></p><p><strong>3.1 Port Distribution Analysis</strong></p><pre><code>PortScanCICIDS_CL
| summarize count() by Destination_Port_d
| sort by count_ desc</code></pre><p>This query groups all connection records by destination port and counts how many times each port was hit, then sorts from most to least. In normal network traffic, you would see a handful of ports with high counts (like port 443 for web browsing). Seeing hundreds of different ports each with connection attempts is a clear indicator of scanning - the attacker is methodically trying every port to see which ones respond.</p><p><img src="media/port-scan-detection-report-2.webp" /></p><table><thead><tr><th><p><strong>Port</strong></p></th><th><p><strong>Hits</strong></p></th><th><p><strong>Protocol</strong></p></th><th><p><strong>Significance</strong></p></th></tr></thead><tbody><tr><td><p>80</p></td><td><p>124</p></td><td><p>HTTP</p></td><td><p>Hunting for web servers</p></td></tr><tr><td><p>443</p></td><td><p>85</p></td><td><p>HTTPS</p></td><td><p>Hunting for secure web servers</p></td></tr><tr><td><p>21</p></td><td><p>69</p></td><td><p>FTP</p></td><td><p>Hunting for file transfer services</p></td></tr><tr><td><p>22</p></td><td><p>68</p></td><td><p>SSH</p></td><td><p>Hunting for remote access entry points</p></td></tr><tr><td><p>444</p></td><td><p>54</p></td><td><p>HTTPS-alt</p></td><td><p>Alternative secure web port</p></td></tr></tbody></table><p>The attacker did not scan randomly - they prioritized well-known service ports. This suggests deliberate intent to find exploitable services rather than generic noise.</p><p><strong>3.2 Unique Ports Scanned</strong></p><pre><code>PortScanCICIDS_CL
| summarize UniquePortsScanned = dcount(Destination_Port_d)</code></pre><p>This query counts the total number of distinct destination ports that appeared in the dataset using dcount (distinct count). Unlike the previous query which showed per-port counts, this gives a single number representing the full scope of the scan. Normal business applications communicate on 1 to 5 known ports. A result of 998 unique ports is the clearest possible evidence of systematic port scanning.</p><p><img src="media/port-scan-detection-report-3.webp" /></p><p>998 unique ports were scanned - the attacker mapped nearly the entire port range.</p><p><strong>3.3 Flow Duration Analysis</strong></p><pre><code>PortScanCICIDS_CL
| summarize AvgDuration = avg(Flow_Duration_d), MaxDuration = max(Flow_Duration_d)</code></pre><p>This query calculates the average and maximum duration of all network flows in microseconds. Flow duration tells us how long each connection attempt lasted. A real user browsing a website or transferring a file would maintain a connection for several seconds or minutes. An attacker using a port scanner sends a probe packet and immediately moves on - resulting in extremely short durations. The 28ms average here confirms no real sessions were established; this was purely automated probing.</p><p><img src="media/port-scan-detection-report-4.webp" /></p><table><thead><tr><th><p><strong>Metric</strong></p></th><th><p><strong>Value</strong></p></th><th><p><strong>Interpretation</strong></p></th></tr></thead><tbody><tr><td><p>Average Duration</p></td><td><p>28ms</p></td><td><p>Automated probe - no real session</p></td></tr><tr><td><p>Maximum Duration</p></td><td><p>34 seconds</p></td><td><p>Occasional port that responded</p></td></tr></tbody></table><p><strong>4. MITRE ATT&amp;CK Mapping</strong></p><table><thead><tr><th><p><strong>Field</strong></p></th><th><p><strong>Value</strong></p></th></tr></thead><tbody><tr><td><p>Tactic</p></td><td><p>TA0007 - Discovery</p></td></tr><tr><td><p>Technique</p></td><td><p>T1046 - Network Service Discovery</p></td></tr><tr><td><p>Evidence</p></td><td><p>998 ports probed at 28ms avg per probe</p></td></tr><tr><td><p>Kill Chain Phase</p></td><td><p>Reconnaissance - attacker mapping attack surface before exploitation</p></td></tr></tbody></table><p><strong>What this means:</strong> T1046 is always a precursor. Attackers don't scan ports for fun - they do it to find open services they can exploit next. Seeing this technique means the attacker was in the early reconnaissance phase and likely planned a follow-on attack targeting the services they discovered open (HTTP, HTTPS, FTP, SSH).</p><p><strong>5. Recommendations</strong></p><p><strong>5.1 Immediate Actions</strong></p><ul><li>Block the source IP at the perimeter firewall</li><li>Review logs for any follow-on access attempts on ports 80, 443, 21, and 22</li><li>Check whether any scanned ports successfully responded and if further exploitation occurred</li></ul><p><strong>5.2 Sentinel Detection Rule</strong></p><pre><code>PortScanCICIDS_CL
| summarize PortsScanned = dcount(Destination_Port_d) by bin(TimeGenerated, 1m)
| where PortsScanned &gt; 100
| extend AlertSeverity = "Medium", Technique = "T1046"</code></pre><p>This detection rule runs continuously in Sentinel and groups connection records into 1-minute time windows. If any single minute contains more than 100 unique destination ports being hit, it fires an alert. The 1-minute window is important - it catches fast automated scanners that try to complete their scan quickly before detection.</p><p><strong><img src="media/port-scan-detection-report-5.png" /></strong></p><p><strong>5.3 Long-term Hardening</strong></p><ul><li>Disable FTP (port 21) - replace with SFTP which encrypts credentials</li><li>Restrict SSH (port 22) access to known IP ranges via firewall rules</li><li>Enable rate limiting on inbound connections to slow future scan attempts</li><li>Deploy Suricata IDS for real-time network scan detection</li></ul><p><strong>6. Conclusion</strong></p><p>The investigation confirmed a deliberate and automated port scan attack. A total of 998 unique destination ports were probed across 5,000 connection flows with an average duration of just 28 milliseconds per probe. The attacker specifically targeted well-known service ports - HTTP (80), HTTPS (443), FTP (21), and SSH (22) - indicating clear intent to discover exploitable services for a follow-on attack. This activity was successfully detected using Microsoft Sentinel KQL queries and maps to MITRE ATT&amp;CK T1046. Immediate firewall blocking and deployment of the Sentinel detection rule are recommended to prevent recurrence.</p>
