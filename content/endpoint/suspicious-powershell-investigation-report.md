(32-bit)</li><li><strong>Parent Process:</strong> C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe (64-bit)</li><li><strong>Flags used:</strong> -noni -nop -w hidden -c</li></ul><p>The command line contained a Base64-encoded, Gzip-compressed payload passed via FromBase64String and GzipStream.</p><p><strong>Red flags identified:</strong></p><ul><li>-w hidden : window hidden from user view</li><li>-nop : bypasses PowerShell profile restrictions</li><li>-noni : non-interactive, no user prompts</li><li>PowerShell spawning a child PowerShell process</li><li>Encoded and compressed src="media/suspicious-powershell-investigation-report-3.png" /></p><p><strong>Step 2 : Payload Decoding</strong></p><p>The Base64 + Gzip encoded payload was decoded using Python:</p><p>import base64, gzip</p><p>decoded = gzip.decompress(base64.b64decode(data))</p><p><strong>Decoded script revealed:</strong></p><p>The payload used Windows API calls via reflection to:</p><ol><li>Call VirtualAlloc : allocate executable memory in RAM</li><li>Call Marshal.Copy : copy raw shellcode bytes into allocated memory</li><li>Call CreateThread : execute the shellcode in a new thread</li><li>Call WaitForSingleObject : keep the process alive</li></ol><p>This is a <strong>Metasploit reverse shell</strong> delivered as a reflective shellcode injection. The shellcode runs entirely in memory with no files written to disk - a fileless attack technique.</p><p><img src="media/suspicious-powershell-investigation-report-4.png" /></p><p><strong>Step 3 : Attack Chain (Event 4688 Timeline)</strong></p><p>A review of all process creation events on srvdefender01.offsec.lan between 09:56 and 10:19 UTC revealed the following attack chain:</p><p>services.exe</p><p>↓</p><p>cmd.exe</p><p>↓</p><p>powershell.exe (64-bit) -&gt; runs encoded Metasploit payload</p><p>↓</p><p>powershell.exe (32-bit) -&gt; injects shellcode into memory</p><p>↓</p><p>cscript.exe</p><p>↓</p><p>conhost.exe</p><p>The attack originated from services.exe, indicating the attacker likely had existing access and used a Windows service to trigger the attack chain.</p><p><strong>Step 4 : Network Activity</strong></p><p>Event ID 3 (Sysmon network connection) was queried for the affected machine. No results were returned - Sysmon network logging was not enabled on this endpoint or not included in the available dataset.</p><p><strong>Step 5 : Persistence Check</strong></p><p>Queries for Event ID 4698 (scheduled task created) and Event ID 13 (Sysmon registry modification) returned no results.</p><p>This is consistent with a fileless attack - the Metasploit shellcode ran entirely in memory and did not require traditional persistence mechanisms such as registry run keys or scheduled tasks.</p><p><strong>4.
